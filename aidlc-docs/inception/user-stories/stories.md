# 사용자 스토리

## 스토리 작성 방식

- **분해 방식**: 페르소나 기반 + 사용자 여정 기반 hybrid.
- **페르소나**: 고객, 매장 관리자.
- **Acceptance criteria 형식**: 체크리스트.
- **상세 수준**: 프로토타입에 맞춘 간결한 핵심 흐름.
- **우선순위**: 모든 스토리를 MVP 필수 범위로 본다.
- **추적성**: 요구사항 영역을 스토리 그룹 단위로 매핑한다.

## 고객 스토리

관련 요구사항 영역: FR-CUST-001~FR-CUST-005, UX 및 접근성, 성능, 신뢰성 및 복원력.

### US-CUST-001: 테이블 자동 로그인

고객으로서, 별도 로그인 없이 주문을 시작할 수 있도록 테이블 태블릿이 테이블 설정을 기억하길 원한다.

Acceptance criteria:

- [ ] 고객은 초기 설정에서 매장 식별자, 테이블 번호, 테이블 비밀번호를 입력할 수 있다.
- [ ] 성공한 설정은 로컬에 저장된다.
- [ ] 태블릿에 다시 접근하면 테이블 context가 자동 복원된다.
- [ ] active table session ID를 주문 생성과 주문 내역 조회에 사용할 수 있다.

### US-CUST-002: 카테고리별 메뉴 탐색

고객으로서, 원하는 메뉴를 빠르게 찾기 위해 메뉴를 카테고리별로 탐색하고 싶다.

Acceptance criteria:

- [ ] 테이블 context가 준비되면 메뉴 화면이 기본 고객 화면이다.
- [ ] 메뉴 항목은 카테고리별로 표시된다.
- [ ] 각 항목은 이름, 가격, 설명, 카테고리를 표시한다.
- [ ] 고객은 카테고리 간 빠르게 이동할 수 있다.
- [ ] 고객 조작 요소는 터치 친화적이다.

### US-CUST-003: 장바구니 관리

고객으로서, 주문 전 내용을 확인할 수 있도록 메뉴를 장바구니에 담고 수량을 조절하고 싶다.

Acceptance criteria:

- [ ] 고객은 메뉴 항목을 장바구니에 추가할 수 있다.
- [ ] 고객은 장바구니 항목을 삭제할 수 있다.
- [ ] 고객은 수량을 증가/감소할 수 있다.
- [ ] 항목 또는 수량 변경 후 장바구니 총액이 즉시 갱신된다.
- [ ] 고객은 장바구니를 비울 수 있다.
- [ ] 장바구니 내용은 localStorage로 새로고침 후에도 유지된다.

### US-CUST-004: 주문 제출

고객으로서, 선택한 항목을 매장에 전달하기 위해 장바구니를 주문으로 확정하고 싶다.

Acceptance criteria:

- [ ] 고객은 제출 전 주문 내용을 검토할 수 있다.
- [ ] 주문 제출에는 매장, 테이블, 세션, 항목, 수량, 단가, 총액 데이터가 포함된다.
- [ ] 주문 성공 시 주문 번호를 표시한다.
- [ ] 주문 성공 시 장바구니를 비운다.
- [ ] 주문 성공 시 5초 후 메뉴 화면으로 이동한다.
- [ ] 주문 실패 시 에러를 표시하고 장바구니는 유지한다.

### US-CUST-005: 현재 세션 주문 내역 조회

고객으로서, 내 테이블이 주문한 내용을 확인하기 위해 현재 테이블 세션의 주문만 보고 싶다.

Acceptance criteria:

- [ ] 고객은 현재 세션 주문 내역을 열 수 있다.
- [ ] 주문은 주문 시각 순으로 정렬된다.
- [ ] 각 주문은 주문 번호, 주문 시각, 항목, 수량, 금액, 상태를 표시한다.
- [ ] 이전 손님의 주문은 제외된다.
- [ ] 완료된 테이블 이용 이력은 현재 세션 화면에서 제외된다.

## 매장 관리자 스토리

관련 요구사항 영역: FR-ADMIN-001~FR-ADMIN-004, 데이터 요구사항, 성능, 신뢰성 및 복원력.

### US-ADMIN-001: 관리자 로그인

매장 관리자로서, 프로토타입 관리자 대시보드에 접근하기 위해 단순한 매장 인증으로 로그인하고 싶다.

Acceptance criteria:

- [ ] 관리자는 매장 식별자와 비밀번호를 입력할 수 있다.
- [ ] 유효한 인증 정보는 관리자 대시보드를 연다.
- [ ] 잘못된 인증 정보는 명확한 오류를 표시한다.
- [ ] OAuth와 2FA 같은 복잡한 인증은 제외한다.

### US-ADMIN-002: 실시간 주문 모니터링

매장 관리자로서, 수동 새로고침 없이 영업을 관리하기 위해 테이블별 신규 주문을 실시간으로 보고 싶다.

Acceptance criteria:

- [ ] 대시보드는 테이블별 카드를 그리드로 표시한다.
- [ ] 대시보드는 SSE로 주문 이벤트를 구독한다.
- [ ] 정상 로컬 실행 조건에서 신규 주문은 2초 이내 표시된다.
- [ ] 각 테이블 카드는 테이블 번호, 현재 세션 총액, 최신 주문 미리보기를 표시한다.
- [ ] 신규 주문은 시각적으로 강조된다.
- [ ] 대시보드는 테이블 필터링을 지원한다.

### US-ADMIN-003: 주문 상세 확인 및 상태 변경

매장 관리자로서, 준비 진행 상황을 추적하기 위해 주문 상세를 확인하고 상태를 변경하고 싶다.

Acceptance criteria:

- [ ] 관리자는 대시보드에서 테이블 또는 주문 상세를 열 수 있다.
- [ ] 상세 화면은 전체 메뉴 항목, 수량, 주문 번호, 주문 시각, 총액을 표시한다.
- [ ] 관리자는 주문 상태를 대기중/준비중/완료로 변경할 수 있다.
- [ ] 상태 변경은 주문 상세와 현재 테이블 상태에 반영된다.

### US-ADMIN-004: 테이블 세션 관리

매장 관리자로서, 다음 손님이 깨끗한 세션으로 시작할 수 있도록 테이블 이용 완료를 처리하고 싶다.

Acceptance criteria:

- [ ] 관리자는 테이블 번호와 테이블 비밀번호를 설정할 수 있다.
- [ ] 첫 주문은 table session을 생성하거나 활성화한다.
- [ ] 관리자는 확인 후 테이블 이용 완료를 처리할 수 있다.
- [ ] 이용 완료 시 현재 세션 주문은 과거 이력으로 조회 가능하다.
- [ ] 이용 완료 시 현재 주문 목록과 총액은 0으로 초기화된다.
- [ ] 새 손님은 이전 세션 주문을 볼 수 없다.

### US-ADMIN-005: 주문 정정

매장 관리자로서, 테이블 총액을 정확히 유지하기 위해 잘못된 주문을 확인 후 삭제하고 싶다.

Acceptance criteria:

- [ ] 관리자는 특정 주문 삭제를 요청할 수 있다.
- [ ] 시스템은 삭제 전 확인을 표시한다.
- [ ] 확인된 삭제는 주문을 제거한다.
- [ ] 삭제 후 테이블 총액이 재계산된다.
- [ ] 관리자는 성공 또는 실패 피드백을 받는다.

### US-ADMIN-006: 과거 주문 조회

매장 관리자로서, 특정 테이블의 과거 주문을 확인하기 위해 완료된 테이블 세션 이력을 보고 싶다.

Acceptance criteria:

- [ ] 관리자는 테이블별 과거 주문을 열 수 있다.
- [ ] 과거 주문은 최신순으로 표시된다.
- [ ] 각 항목은 주문 번호, 주문 시각, 메뉴, 총액, 테이블 이용 완료 시각을 포함한다.
- [ ] 관리자는 날짜로 과거 주문을 필터링할 수 있다.
- [ ] 관리자는 이력 화면을 닫고 대시보드로 돌아갈 수 있다.

### US-ADMIN-007: 메뉴 관리

매장 관리자로서, 고객용 메뉴가 현재 판매 메뉴를 반영하도록 메뉴 항목을 관리하고 싶다.

Acceptance criteria:

- [ ] 관리자는 카테고리별 메뉴를 볼 수 있다.
- [ ] 관리자는 이름, 가격, 설명, 카테고리를 가진 메뉴를 생성할 수 있다.
- [ ] 관리자는 메뉴를 수정할 수 있다.
- [ ] 관리자는 메뉴를 삭제할 수 있다.
- [ ] 관리자는 메뉴 표시 순서를 조정할 수 있다.
- [ ] 필수 필드와 가격 범위를 검증한다.

## INVEST 검토

- **Independent**: 스토리는 사용자 흐름별로 나뉘어 집중 구현이 가능하다.
- **Negotiable**: 내부 구현을 과하게 고정하지 않고 사용자 가치를 설명한다.
- **Valuable**: 각 스토리는 고객 또는 관리자에게 보이는 가치를 갖는다.
- **Estimable**: MVP 요구사항 기준으로 범위가 명확하다.
- **Small**: 프로토타입 구현 단위로 다루기 충분히 작다.
- **Testable**: 각 스토리는 체크리스트형 acceptance criteria를 포함한다.
